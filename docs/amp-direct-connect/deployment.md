# Deployment notes (AMP direct-connect)

This document describes the **shape** of a typical AMP deployment of
`dune-admin` in direct-connect mode. It captures the operational layout
and the environment-variable contract — not the exact shell commands,
which can change as the build system evolves.

## Topology

A working direct-connect deployment looks like:

```
Host (Proxmox VM, Ubuntu, etc.)
└── AMP (CubeCoders panel)
    └── Podman container — e.g. AMP_MehDune01
        ├── mock-k8s-go shim (Funcom compatibility layer)
        ├── PostgreSQL                — bound to 127.0.0.1:15432
        ├── RabbitMQ (admin broker)   — 127.0.0.1:5672
        ├── RabbitMQ (game broker)    — 127.0.0.1:5673 (AMQPS)
        ├── BattlegroupDirector       — 127.0.0.1:11717
        └── Game-server processes (one per map)
            — visible as plain processes on the host

Outside the container, on the host:
└── dune-admin (Go binary)
    ├── reads PostgreSQL  → 127.0.0.1:15432
    ├── reads / execs Podman  → discovers game procs & log files
    ├── proxies /director/   → 127.0.0.1:11717
    └── serves SPA + API     → 0.0.0.0:9090
```

The single Go binary terminates the HTTP listener, serves the SPA
assets from `./dist/`, exposes the JSON API, and reverse-proxies the
Director. From a remote admin's browser, everything lives behind one
URL.

## Environment-variable contract

`dune-admin` reads its configuration from environment variables (or a
local `.env`). The full set relevant to direct mode:

| Variable | Default | Meaning |
|---|---|---|
| `CONNECTION_MODE` | `direct` | `direct` (no SSH) or `ssh` (tunnel mode) |
| `DB_HOST` | `127.0.0.1` | PostgreSQL host |
| `DB_PORT` | `15432` | PostgreSQL port |
| `DB_USER` | `dune` | PostgreSQL user |
| `DB_PASS` | (empty) | PostgreSQL password |
| `DB_NAME` | `dune` | PostgreSQL database |
| `DB_SCHEMA` | `dune` | PostgreSQL schema |
| `CONTAINER_NAME` | `AMP_MehDune01` | The Podman container name running the game |
| `CONTAINER_USER` | `amp` | The user that owns the container (used as `sudo -i -u <user>` for `podman exec`) |
| `CONTAINER_LOG_PATH` | `/AMP/duneawakening/logs` | Path inside the container where `.log` files live |
| `DIRECTOR_URL` | `http://127.0.0.1:11717` | Battlegroup Director URL (used by reverse proxy and capture exchange discovery) |
| `LISTEN_ADDR` | `:9090` | Where `dune-admin` listens for HTTP |
| `ALLOWED_ORIGINS` | (see code) | CORS allowlist; same-origin to `LISTEN_ADDR` is always allowed |
| `SCRIP_CURRENCY` | `1` | Internal currency ID — Dune-specific, rarely changed |

`SSH_HOST` / `SSH_USER` / `SSH_KEY` are not required in direct mode.

## Filesystem layout (suggested)

```
/opt/dune-admin/
├── dune-admin              # the Go binary
├── .env                    # generated by `-setup` or hand-edited (chmod 600)
├── dist/                   # SPA build output (web/dist/ copied or built here)
├── item-data.json          # static item catalog
└── tags-data.json          # journey-node → tag mappings
```

The binary, when run from this directory, will:

1. Load `.env` from the cwd.
2. Bind to `LISTEN_ADDR`.
3. Serve `./dist/` for any path that isn't an API route or `/director/`.
4. Mount the API at `/api/v1/...`.
5. Mount the Director reverse proxy at `/director/` (direct mode only).

## Running as a service

The natural systemd unit looks like:

```ini
[Unit]
Description=Dune Admin (AMP direct-connect)
After=network-online.target

[Service]
Type=simple
WorkingDirectory=/opt/dune-admin
ExecStart=/opt/dune-admin/dune-admin
Restart=on-failure
# Run as a user that can `sudo -i -u amp podman exec ...` without a TTY.

[Install]
WantedBy=multi-user.target
```

The service account must be able to `sudo -i -u <CONTAINER_USER>
podman exec <CONTAINER_NAME> ...` non-interactively, because that's how
the battlegroup adapter (ADR-002) and log streamer (ADR-003) reach
inside the container. A line in `/etc/sudoers.d/` granting the
service user passwordless `sudo` for that exact command shape is
sufficient — broader sudo grants are not needed.

## Build / deploy at a high level

The deployment is two artifacts:

- The **Go binary**, cross-compiled for the host
  (typically `GOOS=linux GOARCH=amd64`).
- The **SPA bundle** — `web/dist/` produced by the frontend's `npm run
  build`.

A "deploy" copies both to `/opt/dune-admin/`, restarts the systemd unit,
and is done. Database migrations and configuration are not part of a
deploy — `.env` is generated once during initial setup (`./dune-admin
-setup`) and edited in place after.

The exact build / scp / restart sequence belongs in the repo's main
README or in a `scripts/` directory; it's intentionally kept out of
this doc because it changes as the build system evolves and shouldn't
require updating an ADR-style document every time.

## Observability notes

A few things to wire into your monitoring if you operate this:

- **`/api/v1/status`** returns `{connection_mode, ssh_connected,
  db_connected, version, ...}`. Scrape this for a basic liveness
  signal; `db_connected: false` is the most actionable failure.
- **Systemd journal** for the `dune-admin` service is the primary log
  source. Structured logging is not yet implemented — see future
  work below.
- **`/director/`** proxies to the Director, which has its own metrics
  surface — worth scraping if you care about live game-server state.

## Known future work

These are explicitly out of scope for the current adaptation but worth
flagging:

- **Structured logs** with correlation IDs and severity levels (today's
  logging is `log.Printf`-style stdlib output).
- **Metrics endpoint** for Prometheus / OpenTelemetry — the four golden
  signals are not currently surfaced.
- **Authentication** in front of the API and the Director proxy. Clerk
  is wired into the frontend but the API is unauthenticated; in a
  shared-host scenario this is the obvious next gap.
- **TLS termination** — currently expected to be handled by a fronting
  proxy (Cloudflare, Caddy, nginx) if needed.
